1. Introduction & Problem Statement

1.1 Context

Millions of micro-entrepreneurs across India — cobblers, potters (kumhar), tailors, artisans, and small vendors — possess valuable skills honed over generations. However, they lack digital visibility and modern channels to reach customers. Most rely entirely on foot traffic and word-of-mouth, which limits income growth, geographic reach, and long-term sustainability.

HunarHub (Hindi: "हुनर" means skill/craft) is a full-stack web application built to solve this problem. It provides a centralized digital marketplace where micro-entrepreneurs can showcase their skills, sell handmade products, and accept custom service requests — while customers can easily discover and support local talent.

1.2 Problem Statement

The specific challenges faced by micro-entrepreneurs that this platform addresses:

  • No digital presence or marketing reach — HunarHub gives every artisan a public profile with their skills, gallery, and pricing.
  • Limited customer discovery beyond local areas — The marketplace is accessible globally via the internet.
  • No structured way to accept service requests or orders — The platform provides a formal service request system with status tracking.
  • Dependency on middlemen or offline sales — Direct artisan-to-customer connection eliminates intermediaries.
  • Lack of transparency in pricing and availability — Every product and service has clear pricing and stock status.

2. Project Objectives

2.1 Primary Objectives

  1. Digitally connect local micro-entrepreneurs with customers — Achieved through the marketplace, artisan profiles, and product listings.
  2. Enable service booking and product selling from one platform — The platform supports both e-commerce (product orders) and a custom service-request system (tailoring, repairs, commissions).
  3. Promote traditional skills and handmade products — The homepage features artisan categories (Cobbler, Potter, Tailor, Artisan, Small Vendor) and an "Our Story" section highlighting the mission.
  4. Increase income opportunities for small vendors — Artisans have a dedicated dashboard to manage inventory, track orders, and view earnings.

2.2 Secondary Objectives

  1. Encourage local and sustainable commerce — The platform promotes handcrafted, locally-made goods.
  2. Reduce dependency on middlemen — Direct buyer-to-seller transactions with transparent pricing.
  3. Provide entrepreneurs with simple digital tools — The Seller Dashboard is designed to be intuitive with clear navigation and simple forms.
  4. Enable community-based economic growth — As more artisans onboard, the platform becomes a local economic hub.

5. System Architecture

HunarHub follows a decoupled client-server architecture. The React frontend and Node.js backend are completely independent applications that communicate exclusively over HTTPS via RESTful API calls.

USER (Browser)
↕ HTTPS
REACT FRONTEND — Vercel CDN (Static SPA)
↕ Axios HTTP Requests (Authorization: Bearer JWT)
EXPRESS.JS REST API — Render (Node.js Web Service)
↕ Mongoose ODM
MONGODB ATLAS (Cloud Database Cluster)

5.1 Why Decoupled?

  • Independent scaling: Frontend (static files) and backend (compute-intensive API) scale separately.
  • Faster frontend delivery: Vercel serves the React build from edge servers closest to the user.
  • Independent deployments: A CSS fix doesn't require restarting the backend, and vice versa.
  • Team readiness: In a real team, frontend and backend developers work independently.

MongoDB was chosen because e-commerce products have varying attributes (a pottery item has different specifications than a tailored garment), and MongoDB's flexible document model handles this without rigid table migrations.

8.3 Key Design Decisions

  • ObjectId References with .populate(): Instead of duplicating seller data inside every product document, we store only the ObjectId and use Mongoose's populate to resolve references at query time.
  • Compound Unique Index on Reviews: { product: 1, user: 1 } ensures one review per user per product at the database level.
  • Embedded Sub-documents: shippingAddress inside Order is embedded (not referenced) because it's a one-to-one relationship that never needs independent querying.
  • Server-Side Price Calculation: totalPrice = product.price × quantity is always computed on the backend to prevent price manipulation from the client side.

10. Authentication & Security

10.1 Password Handling

Passwords are never stored in plain text. A Mongoose pre('save') hook intercepts every save operation on the User model. If the password field has been modified, it generates a cryptographic salt using bcrypt.genSalt(10) and hashes the password before it reaches the database. The select: false flag on the password field ensures it is excluded from all queries by default.

10.2 JWT Token Flow

  1. User submits login credentials (email + password).
  2. Backend finds the user and uses bcrypt.compare() to validate the password against the stored hash.
  3. If valid, backend signs a JWT: jwt.sign({ id: user._id }, SECRET, { expiresIn: '30d' }).
  4. Token is sent to the frontend and stored in localStorage.
  5. All subsequent requests include: Authorization: Bearer <token>.
  6. The protect middleware extracts, verifies, and decodes the token on every protected route.

10.3 Role-Based Access Control (RBAC)

Two Express middleware functions work in tandem:

  • protect — Verifies the JWT. Attaches the user object to req.user. Blocks unauthenticated requests (401).
  • authorize(...roles) — Checks if req.user.role is in the allowed list. Blocks unauthorized roles (403).

Example: router.get('/analytics', protect, authorize('admin'), handler) — Only authenticated admins can access analytics.

10.4 Seller Approval Workflow

  1. User registers with role = "seller". Their account is created with isApprovedSeller: false.
  2. The seller can log in but cannot list products or accept orders until approved.
  3. An Admin reviews the pending application in the Admin Dashboard.
  4. On approval, isApprovedSeller is set to true — unlocking the full seller dashboard.

11. Frontend Architecture

11.1 State Management

Global state is managed using React's built-in Context API:

  • AuthContext: Stores the logged-in user and JWT. Provides login(), register(), logout(). Restores session from localStorage on page load.

16. Challenges & Solutions

ChallengeSolution
CORS errors when frontend (port 5173) and backend (port 5001) run on different origins Used the cors npm package as Express middleware to allow cross-origin requests.
User session lost on page refresh (React state resets) Persisted JWT token in localStorage. AuthContext reads it on mount and calls /api/auth/me to restore session.
Shopping cart data lost on refresh Synced cart state to localStorage on every change. CartContext reads it on mount.
Price manipulation risk from client-side calculation All price calculations happen server-side. Backend fetches real product price from the database.
Ensuring marketplace quality (preventing spam sellers) Implemented approval workflow: sellers are isApprovedSeller: false until Admin manually approves.
Managing different API URLs for local vs. production Used Vite's import.meta.env.VITE_API_URL with environment-specific .env files.
Safe feature development without breaking production Used Git feature branches (e.g., feature-homepage-scroll), tested locally, then merged to main.

17. Key Learnings

  1. Full-Stack Integration: Understood how frontend and backend are two separate applications communicating via a well-defined API contract.
  2. Authentication is a System, Not a Feature: JWT touches every layer — database (password hashing), middleware (token verification), frontend (token storage), and UI (conditional rendering).
  3. Database Design Matters: Choosing between embedding vs. referencing data in MongoDB has real performance and consistency implications.
  4. Environment Variables Are Essential: Secrets must never be in source code. The .env + .gitignore pattern is non-negotiable.
  5. Deployment Is Part of Development: Setting up Vercel and Render with CI/CD webhooks taught modern DevOps practices.
  6. Git Branching Workflow: Used feature branches for isolated development, then merged to main after testing — mirroring professional team workflows.
  7. CSS Is Powerful Alone: Building the entire UI with vanilla CSS deepened understanding of Flexbox, Grid, custom properties, and responsive design.
